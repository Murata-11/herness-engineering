# 献立計画アシスタントの利用入口と保存方式

状態: accepted

日付: 2026-07-14

決定者: プロジェクトオーナー

見直し: 外部公開、Web UI、外部データ源、依存パッケージ、または保存方式を検討する時

関連: [プロダクト仕様](../product-specs/meal-planning-assistant.md)、[アーキテクチャ](../../ARCHITECTURE.md)、[実装](../../meal_schedule/)、[Codexスキル](../../.agents/skills/plan-meals/SKILL.md)、[実行計画](../exec-plans/completed/2026-07-14-meal-planning-mvp.md)

## 背景

目的には「このプロジェクトでCodexに質問する」という利用像と、献立を永続化して後から確認する要件がある。入口の選択は技術スタック、依存関係、データ形式、検証方法に長く影響するため、MVPの境界を明示する必要がある。

## 決定

MVPは「Codex向けのリポジトリ内スキル + Python標準ライブラリだけで動くローカルCLI + 人が読めるバージョン管理可能なJSONデータ」とする。

スキルが自然文の条件確認、履歴を踏まえたレシピ生成、表示、保存前確認を担う。CLIが設定・履歴の検証、原子的保存、期間検索、重複候補取得、買い物集約を決定的に行う。初回提案は保存せず、利用者が内容を明示的に確定した後だけ保存する。

実設定はGit管理外、確定献立は `data/plans/` に保存する。外部通信、Web UI、データベースは導入しない。

## 検討した選択肢

| 選択肢 | 判断理由 |
| --- | --- |
| Codex向けスキル + ローカルCLI（採用） | 利用像に直接合い、生成はCodex、記録の整合性はテスト可能なコードへ分離できる。追加サービスや実行依存が不要。 |
| ローカルCLIのみ | 実装は単純だが、自然文の依頼、レシピ生成、条件確認を一貫してCodexへ案内する入口が弱い。 |
| Web UI付きアプリ | Codex外から閲覧しやすいが、フレームワーク、API、認証、配備、データ保護の判断が必要になり、MVPの要求を超える。 |
| MarkdownファイルをCodexが直接編集 | 依存がない一方、入力検証、期間検索、重複検出、データ移行を安定して機械検証しにくい。 |

## 影響

- 外部サービスなしで、Codexとの会話から生成・記録・再確認までを利用できる。
- 保存形式とCLI契約を保てば、将来Web UIを追加しても履歴を再利用できる。
- レシピ文章の品質はCodexに依存するため、構造・禁止食材の単純一致・履歴重複など検査可能な部分をCLIで守る。
- ローカル履歴の同期・共有はGit等の利用者側の手段に依存する。
- 実設定をGit管理外にするため、新しい環境では利用者が設定を再作成する必要がある。

## 守り方

- [自動テスト](../../tests/)でCLI契約、保存、検索、買い物集約、失敗条件を検証する。
- [Codexスキル](../../.agents/skills/plan-meals/SKILL.md)に検証と明示確認後保存を必須化する。
- 依存追加、外部通信、機微情報の保存を導入する変更では、この判断を見直す。
