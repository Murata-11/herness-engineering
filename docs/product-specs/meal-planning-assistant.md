状態: approved

関連: [実装](../../meal_schedule/)、[Codexスキル](../../.agents/skills/plan-meals/SKILL.md)、[設計判断](../design-docs/2026-07-14-meal-planning-interface.md)、[実行計画](../exec-plans/completed/2026-07-14-meal-planning-mvp.md)

## 背景と課題

家庭で毎日の献立を考える際には、料理の重複を避けながら好み、調理負担などを考え、レシピと買い物内容まで具体化する必要がある。これらを毎週繰り返す負担があり、過去の献立を後から参照できないと献立の偏りも把握しにくい。

## 目的

利用者がCodexへ自然文で依頼することで、一定期間の献立、各料理のレシピと手順、まとめた買い物リストを得られ、確定した献立を記録して後から確認できるようにする。

## 対象ユーザー

- 家庭の食事を計画する人: 献立決めから買い物準備までの時間を減らし、過去の献立を参考に変化をつけたい。

## スコープ

- 「今週の献立を考えて」のような自然文の依頼から、対象期間の献立案を作る。
- 各食事について料理名、人数分の材料、分量、調理手順を提示する。
- 期間内で必要な食材をまとめ、利用元の料理が分かる買い物リストを提示する。
- 利用者が確定した献立だけを日付とともに永続的に記録する。
- 記録済みの献立を期間または日付で後から確認する。
- 記録済みの献立を参照し、料理の繰り返しを避けた提案に役立てる。
- 人数、食事回数、苦手な食材、アレルギー、予算、調理時間などの条件を利用する。

## 非スコープ

- 食材の購入、店舗価格・在庫のリアルタイム取得、配送サービスとの連携。
- 医療上の診断、治療、または安全保証を目的とした栄養・アレルギー対応。
- 複数利用者のアカウント管理、クラウド同期、公開サービス運用。

## 利用シナリオ

1. 利用者がCodexに「今週の献立を考えて」と依頼する。
2. Codexは保存済みの条件と直近28日の履歴を確認し、不足する重要条件があれば利用者へ確認する。
3. Codexは日ごとの献立、材料と手順、集約した買い物リストを提示する。
4. 利用者が内容を明示的に確定すると、Codexは検証済み献立を保存する。
5. 利用者が後日「先週の献立を見せて」と依頼すると、記録済みの内容が表示される。

## 受け入れ条件

- [x] 対象期間と人数が与えられた場合、各日の献立を日付付きで提示できる。
- [x] 各献立に、人数に対応する材料と、順序の分かる調理手順が含まれる。
- [x] 買い物リストでは同じ食材・単位・カテゴリが集約され、料理ごとの材料との対応を失わない。
- [x] 確定した提案をリポジトリ内の永続データとして記録し、新しいCodexセッションから参照できる。
- [x] 指定した期間の記録を取得し、存在しない場合はその旨を伝えられる。
- [x] 直近の料理名を取得し、スキルが重複を避けるか、重複理由を明示できる。
- [x] アレルギーまたは苦手な食材が設定されている場合、料理名・材料名の単純一致を機械的に拒否する。
- [x] 不正または不完全な記録を読み込んだ場合、黙って誤った献立を返さず、修正可能なエラーを示す。
- [x] 初回提案は自動保存せず、利用者の明示確認後だけ保存する手順がスキルに定義されている。

## 前提と制約

- MVPの入口は、リポジトリ内CodexスキルとPython標準ライブラリだけのローカルCLIとする。
- 設定がない場合、スキルは人数、対象食事、アレルギーの有無を推測せず確認する。
- レシピ文章はCodexが生成し、構造、日付、人数、単純な禁止食材一致はCLIが検証する。
- 派生原料、別名、交差接触などは単純一致では保証できない。曖昧な場合は利用者が確認し、医療上の安全判断には用いない。
- Web UI、外部レシピ情報源、栄養計算、クラウド同期は実利用から必要性が確認できた時に別途判断する。
